⚙️ Setup Journey

Step by step — including every blocker and fix

1 — Clone & fix Node version

Repo cloned from for-tests branch. Terminal still running Node 16 after nvm alias default — had to run nvm use 20.16.0 explicitly in the existing session before yarn would succeed.

# Switch Node in current session — alias alone is not enough
$ nvm use 20.16.0
$ node --version
v20.16.0

2 — Docker: blocker identified

Docker 17.03.1-ce (Docker Toolbox, 2017) cannot run docker-compose.yml format v3.4 which requires Engine 17.09+. The app's in-app updater only shows updates within the same Toolbox branch — Docker Desktop 4.x is a separate product requiring a manual download. No-Docker path adopted instead.

3 — PostgreSQL via Homebrew

Installed PostgreSQL 14 via Homebrew as the database layer. Created the periodtracker database and user manually.

$ brew install postgresql@14
$ brew services start postgresql@14
$ psql postgres -c "CREATE DATABASE periodtracker;"
psql (PostgreSQL) 14.22 (Homebrew) — connected

4 — Environment files

Ran yarn copy-config to generate .env files from .env.dist templates. Then fixed two values Docker-specific configs had set incorrectly for local use:

# packages/api/.env
- DATABASE_HOST=postgres
+ DATABASE_HOST=localhost

# packages/cms/.env
- DATABASE_HOST=postgres
+ DATABASE_HOST=localhost

5 — Submodule bug: "already exists in the index"

yarn modules failed immediately. The remove script clears .gitmodules and .git/modules/ but leaves stale entries in the git index — so git submodule add refuses to re-add them.

fatal: 'app/src/resources' already exists in the index
fatal: 'packages/core/src/common' already exists in the index
# Fix: clear stale index entries first
$ git rm --cached .k8s app/src/resources packages/core/src/common packages/delete-account 2>/dev/null
$ yarn modules
✓ All submodules cloned successfully

6 — Wrong branch: master vs for-tests

Initial setup was done on master. The task requires for-tests. master has static imports of avatar customisation assets (friendAssets, BodyComponents) that only exist in an optional submodule not pulled by yarn modules — Metro crashes on first bundle.

$ git stash
$ git checkout for-tests
$ git stash pop
# Resolved conflicts on deleted GitHub template files
$ git rm .github/ISSUE_TEMPLATE/bug_report.md .github/ISSUE_TEMPLATE/feature_request.md .github/pull_request_template.md

7 — Wrong workspace name for CMS

yarn workspace @oky/cms start fails — the actual workspace name in package.json is @oky/oky-cms. Also, @oky/core must be compiled before the CMS starts or it crashes with "Cannot find module".

# Terminal 1
$ yarn workspace @oky/api watch
Application is now running on port 3000

# Terminal 2 — compile core first
$ yarn workspace @oky/core compile
$ yarn workspace @oky/oky-cms start
Server started on port 5000

8 — Expo Go SDK mismatch

Project uses Expo SDK 52. The Android device had Expo Go SDK 54 installed. Newer Expo Go drops support for older SDK versions — scanning the QR code produced a fatal version error. Downloaded the SDK 52 APK directly from the Expo archive.

Error: This project requires Expo SDK 52, but you are using SDK 54

# Fix: install correct Expo Go version
Downloaded: expo-go-sdk-52-android.apk
Installed via ADB sideload

9 — IP address for physical device

Ngrok tunnel failed (requires registered account). Used direct LAN connection instead — updated app/.env with the local network IP shown in Metro output.

# app/.env
EXPO_PUBLIC_API_BASE_URL=http://192.168.1.16:3000
EXPO_PUBLIC_API_BASE_CMS_URL=http://192.168.1.16:5000

App confirmed running on Android device via Expo Go SDK 52. API on port 3000, CMS on port 5000, PostgreSQL on localhost.

🐛 Bugs & Issues Found in the Repo

Identified during setup — documented in SETUP_LOG.md and PROPOSAL.md

🔴 yarn modules fails on any repo with prior git history

bin/modules/remove.sh clears .gitmodules and .git/modules/ but does not clear the git index. git submodule add then refuses to add paths that are still tracked. This affects every developer who clones the repo because git clone registers submodule paths in the index by default.

✅ Fix: git rm --cached the four submodule paths before running yarn modules. Proposed: fix remove.sh to include this step automatically.
🔴 master branch has static imports of optional assets

USE_AVATAR_CUSTOMIZATION=true is set in .env.dist by default, but the required asset files (friendAssets.ts, BodyComponents.tsx) only exist in an optional submodule. The imports are static — not guarded by any env check — so Metro crashes for any developer who doesn't have that private submodule.

✅ Workaround: Use for-tests branch, which doesn't have these imports. Long term: make the imports conditional or stub the assets in the white-label submodule.
🟡 @oky/core must be compiled before CMS starts — undocumented

The CMS depends on @oky/core but its dist/ folder is not committed. If you start the CMS before compiling core, it crashes with Cannot find module '@oky/core'. Nothing in the docs mentions this step order.

✅ Fix: Run yarn workspace @oky/core compile before yarn workspace @oky/oky-cms start. Documented in SETUP_LOG.md.
🟡 Expo Go SDK version not documented

The project targets Expo SDK 52. The README makes no mention of this requirement. Newer Expo Go versions (SDK 53, 54) silently drop backwards compatibility — the QR scan fails with a version error that first-time contributors would not be able to diagnose.

✅ Fix: Download Expo Go SDK 52 APK from the Expo archive. Proposed: add explicit Expo SDK version notice to README.
🟡 CMS workspace name is @oky/oky-cms not @oky/cms

The docs (and intuition) suggest yarn workspace @oky/cms start, but the actual workspace name in package.json is @oky/oky-cms. The wrong name produces a silent "workspace not found" error.

✅ Fix: Use yarn workspace @oky/oky-cms start. Proposed: document workspace names in README.

🎬 Lottie Animation Swap

Understanding Lottie

Lottie is a JSON-based animation format exported from Adobe After Effects using the Bodymovin plugin. React Native renders them via lottie-react-native. The Oky app drives animations by seeking to specific frame ranges — wave (frames 0–4/30), jump (8–10/30), four dance sequences (10–30/30).
