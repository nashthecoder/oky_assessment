Task 2

Modify the app and CMS so the encyclopedia displays different entries depending on whether a user registered as urban or rural.

Not Started CMS + API + App

📋 What to Build

  1. 1CMS: Add a field to encyclopaedia entries — Urban only, Rural only, or All users.
  2. 2API: Expose the new field in the encyclopaedia endpoint.
  3. 3App: Filter encyclopaedia entries based on the logged-in user's area selection at registration.

🏗️ Approach

CMS changes

  1. 1Content filter options — Added Urban only and Rural only options to contentFilterOptions in packages/cms/src/optional.ts. Fixed a bug where the ?? fallback used [] instead of the local defaults, causing the dropdown to render empty when @oky/core does not export the field.
  2. 2Form data wiring — Added contentFilter and ageRestrictionLevel to the article save payload in subcategoryViewScript.js so values are submitted to the API on create/edit.
  3. 3Modal template fix — The ArticleModal.ejs include in both Subcategory.ejs and Encyclopedia.ejs was not passing contentFilterOptions or ageRestrictionOptions to the partial. Updated both includes to forward the variables so the server-side EJS loop can render the options.
  4. 4UI improvement — modal layout — Before this task could be completed, the Insert Article form had to be restructured. The original layout crammed all fields into a horizontal DataTable-style row, making the Filter field invisible and unusable. The form was rebuilt as a vertical stacked layout using Bootstrap form-row grid — Category/Subcategory on one row, then Article Title, Article Content, and finally Filter/Age Restriction/Live as a 3-column bottom row. See before/after below.
  5. 5DataTable column — Added a Filter column to the article list DataTable, rendering the human-readable label (All / Urban only / Rural only) by looking up the stored integer value against the options array injected as a JSON script tag.

Insert Article — Before & After

The UI improvement was a prerequisite for Task 2. Without it, CMS editors had no way to assign urban/rural targeting to an article.

Before — Filter field hidden / empty

  • All fields crammed into a single horizontal row
  • Filter dropdown rendered with no options (empty)
  • contentFilterOptions & ageRestrictionOptions not passed to the EJS partial
  • optional.ts fallback was ?? [] — wiped defaults when @oky/core didn't export the field

After — Filter field visible & functional

  • Vertical stacked layout — each field has space to breathe
  • Filter dropdown shows All / Urban only / Rural only
  • Both view includes updated to forward contentFilterOptions
  • optional.ts fallback changed to ?? contentFilterOptions to preserve local defaults

🐛 Additional Bug Found — Live Toggle on New Articles

⚠️ Live toggle defaults to false on article creation regardless of user input

When creating a new article, the modal's live checkbox can silently reset to unchecked before the form is submitted. The toggle appears active in the UI but the underlying checkbox state is read as false on POST. Found during Task 2 testing — "Understanding period cramps" was toggled live but saved with live: false in the database.

✅ Immediate fix: Set live = true directly in the database via psql.
💡 Proposed: On modal open for new articles, explicitly set $('#col4TableModal').prop('checked', false) and add a visible label showing current state, so the toggle state is unambiguous before submission.
