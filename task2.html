Task 2

Modify the app and CMS so the encyclopedia displays different entries depending on whether a user registered as urban or rural.

Not Started CMS + API + App

📋 What to Build

  1. 1CMS: Add a field to encyclopaedia entries — Urban only, Rural only, or All users.
  2. 2API: Expose the new field in the encyclopaedia endpoint.
  3. 3App: Filter encyclopaedia entries based on the logged-in user's area selection at registration.

🏗️ Approach

CMS changes

  1. 1Content filter options — Added Urban only and Rural only options to contentFilterOptions in packages/cms/src/optional.ts. Fixed a bug where the ?? fallback used [] instead of the local defaults, causing the dropdown to render empty when @oky/core does not export the field.
  2. 2Form data wiring — Added contentFilter and ageRestrictionLevel to the article save payload in subcategoryViewScript.js so values are submitted to the API on create/edit.
  3. 3Modal template fix — The ArticleModal.ejs include in both Subcategory.ejs and Encyclopedia.ejs was not passing contentFilterOptions or ageRestrictionOptions to the partial. Updated both includes to forward the variables so the server-side EJS loop can render the options.
  4. 4UI improvement — modal layout — The original Insert Article modal did not surface the Filter field prominently; it was hidden at the bottom and partially cut off. Restructured the modal body from a dense single-column layout into a responsive two-column row (Bootstrap form-row grid), grouping Filter, Age Restriction, and Live toggle side by side so all fields are visible at a glance without scrolling. This was necessary to make the content-targeting workflow usable for CMS editors.
  5. 5DataTable column — Added a Filter column to the article list DataTable, rendering the human-readable label (All / Urban only / Rural only) by looking up the stored integer value against the options array injected as a JSON script tag.
