🔍 Question 1 — PR Review

Review the pull request at: bugfix/ios-profile-crash

  1. 1 What the change does and how it does it
  2. 2 Any issues to raise before approving — including anything relevant to an app used by girls aged 10 and above
  3. 3 Whether to approve or request changes, and why

Preparation notes — video script bullets

1. What the change does and how it does it

  • On iPhone, tapping Edit Profile could crash the app — no warning, just kicked out
  • The app was trying to display your profile details before they had fully loaded into memory
  • It reached for the data, found nothing, and didn't know what to do — so it crashed
  • The fix teaches the app to check first: do I actually have this person's details?
  • If yes — opens normally. If not — waits quietly instead of crashing
  • User stays inside the app either way

2. Issues to raise before approving

  • The fix shows a blank screen when data isn't ready — safe, but silent
  • For a girl aged 10 tapping Edit Profile and seeing nothing, blank = broken. She doesn't know if the app is loading, frozen, or has lost her data
  • A loading indicator or a gentle "just a moment" message would be far less alarming for this age group
  • The crash is fixed — but why the data arrives null isn't explained or addressed. If it's a logout race condition, it could surface elsewhere in the app
  • No mention of testing on an actual device against the specific crash scenario — important for an iOS-specific bug

3. Approve or request changes — feedback to the dev

  • Removing the as User cast is correct — it was hiding a real null from TypeScript and masking the bug
  • The null guard in useMemo and the early return are clean and right
  • The fallback state in useReducer is dead code — the early return fires first so those values are never used. Remove or comment it
  • Replace return null with a loading spinner — returning nothing is not appropriate UX for this audience
  • Raise a follow-up ticket: what causes the null? Race condition on logout? That underlying issue needs its own fix
  • Decision: request changes — the crash fix is correct and should go in, but not before the blank screen is replaced with a loading state and the dead code is cleaned up
